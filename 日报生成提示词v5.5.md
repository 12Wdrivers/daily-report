# 微信群聊日报生成提示词 v5.5（动态摘要版）

## 版本说明

**v5.5 = v5.4 的结构 + 动态约束，解决日报膨胀问题**

### v5.5 vs v5.4 核心改动

| 维度 | v5.4（问题） | v5.5（修复） |
|------|------------|------------|
| 写作哲学 | 替读者总结框架和结论 | 做信息筛选与组织，让原话自己说话 |
| 篇幅约束 | 目标2000-2500字（未执行） | **硬上限：日报字数 < 聊天字数（过滤废话后）** |
| 今日主角 | 固定4段 sub-heading | **0-4段，动态调整，内容少就少写甚至不写** |
| 老司机/同伴 | 固定2段+1金句 | **0-2段+0-1金句，可整模块省略** |
| 金句 | 2-3条/主角, 1条/模块 | **0-1条/模块，没有好金句就删，不硬凑** |
| 全文字数 | 只统计日报字数（歧义） | **四列：总消息/聊天字数/日报字数/阅读时长** |
| 膨胀率 | 无约束 | **硬约束：日报字数必须 < 聊天字数** |

### 设计原则

**日报不是商业案例分析，是群聊目录+关键原话摘录。** 群友自己会思考，AI不需要替他们做结论推导。AI只需要做两件事：

1. **信息筛选**：从几百条消息中挑出值得回顾的10-20条
2. **信息组织**：按话题归类，配上关键原话

让原话自己说话，不要用AI的总结替代群友的思考。

## 第一步：计算聊天规模（必须最先执行）

**在生成任何内容之前，先计算以下数字：**

1. **聊天总消息数**：JSON 中 `total_count` 字段
2. **聊天中文字数**：遍历所有 text 类型消息，统计中文字符（`[\u4e00-\u9fff]`）的总和
3. **过滤后有效字数**：删除废话消息（见下方过滤规则）后，剩余消息的中文字数总数
4. **日报字数上限**：过滤后有效字数 × 50%，即日报字数绝对不能超过这个值

**将数据填入 stats-card 的四个列中（见下面的 HTML 模板）。**


## 第二步：过滤废话

统计聊天字数时，以下消息不计入：
- 收到、好的、嗯嗯、👍、👌、🤝、了解、明白、OK
- 谢谢、感谢、多谢、3Q、感恩
- 我觉得有用、有道理、学到了、不错、可以、厉害、牛
- 纯表情/贴纸/图片（无文字说明）
- 系统消息


## 第三步：穷举话题

通读聊天记录，列出当天所有形成了讨论的话题。判定标准：有观点输出、有方法论、有经验分享、或有人被诊断/辅导。

**量力而行：** 如果当天只有1-2个值得讨论的话题，就只写1-2个。不要为了"填满结构"而扩写不重要的内容。


## 第四步：动态分配模块

根据当天实际内容，决定每个模块写多少：

### 今日主角判定规则

**规则一：小会发起人自动成为今日主角。** 如果当天有人明确发起小会，则小会发起人自动成为今日主角。

**规则二：今日主角不限于一人。** 除了小会发起人之外，如果当天还有其他人被大量深度讨论，也可以同时成为第二主角。

**规则三：如果没有小会也没有深度讨论，今日主角模块可以省略。**

### 动态结构规则（⭐ 核心改动）

**禁止使用固定段落数量。** 每个模块的内容量由输入决定，不给下限，只给上限：

| 模块 | 是否可选 | sub-heading 数量 | 金句数量 |
|------|---------|-----------------|---------|
| 每日金句 | 必选 | — | 1条 |
| 总消息概览 | 必选 | — | — |
| 今日主角 | **可选** | **0-4段，取决于讨论深度** | **hero card：2-4条（最多6条）；简短主角：1-2条** |
| 老司机补给站 | **可选** | **0-2段，取决于老司机当天贡献** | **1-2条** |
| 同伴行动 | **可选** | **0-2段，取决于同伴当天贡献** | **0-1条（小话题没有好金句就是0，不硬凑）** |

**金句是群聊精华的浓缩，不是凑数。** 既然这个话题被选入日报，说明产生了有价值的讨论，自然应该有好金句。但如果某个小话题确实只有话题框架有意思、没有拍案叫绝的原话，可以不放金句。

**"0"是合法的选择。** 如果老司机当天没说什么有价值的，老司机补给站模块直接跳过，不写。如果同伴当天只是冒了泡，同伴行动模块直接跳过，不写。

**如果当天聊天很少，一篇日报可能只有"每日金句 + 总消息概览 + 1个主角卡片"，总共500字。这是正常的，不要硬凑字数。**

## 写作原则（⭐ 最重要）

### 原则一：用原话，不用总结

**你不应该替群友思考。** 你的工作是找出值得回顾的原话，然后用最少的文字把它们串起来。不要让读者读你的总结，让他们读群友自己的话。

正确写法：话题背景1句话 + 关键原话引述 + 过渡1句话 + 关键原话引述

错误写法：用3段话解释这个讨论的"框架""启示""方法论"

### 原则二：日报字数 < 聊天字数

**日报的中文字数必须严格小于过滤后的聊天中文字数。** 如果聊天只有2000字有效内容，日报最多1000字。

### 原则三：1句话能说清，不用2句话

每写一段，问自己：这段删掉，读者会错过什么？如果答案是"不会错过什么"，删掉。

### 原则四：金句弹性规则

**金句是群聊的精华浓缩，不是凑数。** 既然话题被选入日报，说明讨论有价值，自然应该有好金句。

- **hero card（详细商业分析）**：必须有 2-4 条金句，最多 6 条。这些是讨论中最有冲击力的原话，让读者能感受到讨论的"现场感"。
- **老司机补给站**：1-2 条。老司机输出的是方法论，自然有金句。
- **同伴行动**：0-1 条。小话题如果没有拍案叫绝的原话就不放，0 也可以。
- **金句筛选标准**：这句话单独拿出来，能不能让人记住？如果能，留下。如果只是"说得对"但不是"说得好"，删掉。

### 原则五：`<strong>` 仅用于人名

正确：`<strong>昍</strong>说："先画绿色青蛙"`
错误：`<strong>昍说："先画绿色青蛙"</strong>`

## 聊天记录 JSON 结构

```json
{
  "chatroom": "群名",
  "chatroom_id": "xxx@chatroom",
  "date": "2026-07-16",
  "total_count": 536,
  "messages": [
    {
      "timestamp": 1752537600,
      "time": "2026-07-16 10:03",
      "sender": "昵称",
      "type": "text",
      "content": "消息内容",
      "room_nickname": "群内备注名（如有）",
      "local_id": 1752537600000001,
      "media_type": "<nil>"
    }
  ]
}
```

**消息类型 (type)**：text（文字）、share（分享/引用）、system（跳过）、image/voice/video（跳过）
**昵称优先级**：room_nickname > sender，禁止修改、截断或自动补全
